    # 跨域解决方案

    跨域（非同源策略请求）. 前端/后台在同一个 web 服务器上：同源策略。

    ajax 出现，前后端分离，真实开发中存在跨域。

    协议，域名，端口号，三者一样就是同源，有一样不同，就是跨域。

    非同源的站点的一些限制：

    • 不能够读取和修改对方的 DOM。
    • 不能够访问对方的 Cookie、IndexDB 和 LocalStorage
    • 限制 XMLHttpRequest 请求。

    # JSONP

    通过同源策略涉及不到的"漏洞", 资源文件请求 script，img，link，iframe 加载都不会被同源策略限制到。不存在跨域请求的限制。

    前端通过动态创建 script标签，以callback=func 的形式传给后端，后端通过 query 拿到 callback，组装数据，JSON.stringify(data), 传递给前端。

    浏览器拿到结果，就会把这个函数执行，函数执行（必须是放在全局下面的函数），就可以获取到 data 数据了。

    var script = document.createElement('script');
script.src = 'http://xxx.xxxx.com/list?callback=func';
document.head.appendChild(script);
function func(data) {

}

// 后段
解析 callback拿到 func
组装: `func(${JSON.stringify(data)})`;

    利用 jquery

    $.ajax({
    url: 'http://xxx.xxxx.com/list',
    method: 'get',
    dataType: 'jsonp',
    success: res => {};
})

后端： 
jquery 会帮助我们创建一个不带缓存的函数。
组装之后： 浏览器拿到结果，就会把这个函数执行，会执行 success 函数，拿到 data 数据。

    缺点：

    1. 资源文件请求，jsonp 只能处理 get 请求。
    2. url 传给服务器，在过程中可能发生 url 劫持， 安全性不是很好。
    3. 需要后端的配合。

    # CORS 跨域资源共享

    允许浏览器向跨域的服务器发松 XMLHttpRequest 请求。

    它需要浏览器和服务器同步支持。对于开发人员来说与普通发送请求没什么区别。浏览器一旦发现 AJAX 请求跨域了，就会自动添加一些附加的头信息，有时候还会多出一次附加的请求。只要服务器实现了 CORS 接口，就可以跨域通信。

    CORS 请求分为两类： 简单请求 和 非简单请求

    # 简单请求

    满足一下两大条件，就属于简单请求

    1. 请求方法是： HEAD、GET、POST
    2. HTTP 的头信息是： Accept、Accept-Language、Content-Language、Last-Event-ID、Content-Type: 只限于三个值application/x-www-form-urlencoded、multipart/form-data、text/plain。

    对于非简单请求，浏览器会采用新的处理方式。

    请求过程：

    浏览器直接发出 CORS 请求, 会在头部添加 Origin字段.
服务器根据 Origin 来判断本次请求的是哪个域。服务器根据这个值，决定是否同意这次请求。

如果不在范围内，服务器返回一个正常的 HTTP 响应。浏览器发现头部没有 Access-Control-Allow-Origin 
字段。就知道出错了从而抛出一个错误，被XMLHttpRequest的onerror回调函数捕获.

如果 Origin 在范围内，服务器返回的响应，头部会多出几个东西。
Access-Control-Allow-Origin 可以是 * , 也可以是任意的域名。
Access-Control-Allow-Credentials 布尔值，表示是否可以发送 Cookie。
Access-Control-Expose-Headers  XMLHttpRequest只能拿到六个字段，要想拿到其他的需要在这里指定. 可以通过 xml 对象的 getResponseHeader() 方法拿到 Cache-Control、Content-Language、Content-Type、Expires、Last-Modify、Pragma. 

withCredentials 属性。想要包含 Cookie ，开发者也需要在 AJAX 请求中打开 withCredentials 属性。
对于有的浏览器，有的还是会发送的，就需要显示的关闭。

注意：如果发送 Cookie，Origin 不能设置为 * 号。 Cookie 依然遵循同源政策，只有服务器设置的域才能够上传。且读取不到 Cookie。主要是为了安全性考虑。

    # 非简单请求

    非简单请求，会在正式通信之前，增加一次 HTTP 查询请求，称为预检请求。浏览器会先询问服务器，当前网页所在的域名是否在许可范围中，以及可以使用哪些HTTP 动词和头信息字段，只要又了答复，浏览器才会发出正式的 XMLHttpRequest 请求。这样是为了防止新增的请求，对传统的没有 CORS 支持的服务器形成压力。服务器一个提前拒绝的机会，这样可以防止服务器大量收到DELETE和PUT请求.

    请求方法是PUT或DELETE，或者Content-Type字段的类型是application/json. 是非简单请求。

    “预检”请求用的请求方法是OPTIONS, 表示这个请求是用来询问的.

    “预检”请求的头信息包括两个特殊字段:

    1. Access-Control-Request-Method 表示浏览器的 CORS 请求用到哪些请求方法。
    2. Access-Control-Request-Headers 指定浏览器 CORS 请求会额外发送的头信息字段.

    预检请求的回应:

    1. 服务器通过检查 Origin、method、headers 之后，确定允许跨域请求，可以做出回应。
    2. 如果服务器否定了“预检”请求, 会返回一个正常的 HTTP 回应, 但是没有任何 CORS 相关的头信息字段.
    3. 浏览器就会认定, 服务器不同意预检请求, 触发一个错误.
    4. 服务器回应的其他 CORS 相关字段如下。
      • Access-Control-Allow-Methods: 表示服务器支持的所有跨域请求的方法，避免多次“预检”请求。
      • Access-Control-Allow-Headers: 浏览器请求包括Headers字段, 表明服务器支持的所有头信息字段.
      • Access-Control-Allow-Credentials: Cookie
      • Access-Control-Max-Age: 指定预检请求的有效期。在此期间，不用发出另一条预检请求。
    5. 服务器通过预检请求，之后每次浏览器正常的 CORS 请求，就都跟简单请求一样。会有一个Origin头信息字段。服务器响应 Access-Control-Allow-Origin阻断。

    通过 webpack-dev-server node 服务器帮助我们启动一个服务器，dev-server 也是 node 写的，会在中间价里帮助我们处理请求跨域处理。

    # nginx 反向代理

    反向代理拿到客户端的请求，将请求转发给其他的服务器。主要的场景是用来维护服务器集群的负载均衡。

    比如现在有一个域名 a.com, 服务器的域名为 b.com, 客户端向服务器发送 Ajax 请求。就会产生跨域，这个时候就需要使用 Nginx 了。

    server {
    listen 80;
    server_name a.com;
    location /api {
        proxy_pass b.com;
    }
}

    Nginx 相当于是一个跳板机，跳板机的域名也是 a.com , 让客户端访问 a.com/api, 不会产生跨域的，然后 Nginx 服务器作为反向代理，将请求转发给 b.com, 当响应返回时将响应给到客户端。

    # postMessage

    A 窗口和 B 窗口进行通信发送数据, 把 B 页面嵌入到 A 页面中， 监听 B（iframe）onload 事件，

    A页面
iframe.contentWindow.postMessage('数据', 'http://xxxx');
window.onmessage = function (ev) {
    ev.data;
}

B页面接收 A页面发来的数据
window.onmessage = function (ev) {
    ev.source // A页面
    ev.source.postMessage(ev.data, '', '*');
}

    B 窗口进行 message 监听，监听到了以同样的方式返回数据。

    A 窗口监听到 message， 在进行操作。
