    # 阻塞

    # css 阻塞

    只有 link 外部进来的 css 才能阻塞。

    1. style 标签中的样式。

      • 是由 html解析器进行解析的。
      • 不会阻塞浏览器渲染. (可能会发生白屏)
      • 不阻塞 DOM 的解析。

    2. link 引入的外部 css 样式。

      • 由 css 解析器执行。
      • 阻塞浏览器渲染。 (避免白屏).
      • 阻塞 js 语句的执行。假如后续的 js 会进行读写样式，不阻塞代表 js 可能读取到错误的信息。
      • 浏览器不知道后续的 js 具体内容，只好等到样式都解析完毕后，执行 js。
      • 但是现代浏览器，像 webkit 内核的浏览器而言，仅当脚本尝试访问样式的属性才会禁止脚本。 firfox 是直接禁止脚本的。
      • 不阻塞 DOM 的解析，DOM 解析和 CSS 解析是两个并行的进程, 两棵树最终合并为 渲染树，并行执行的，不阻塞 DOM 的解析。

    3. 优化策略

      • 使用 CDN 节点进行外部资源加速。
      • 对 CSS 进行压缩。
      • 减少 http 请求数，对 css 文件合并。
      • 优化样式表的代码。

    # js 阻塞

    1. 阻塞 DOM 的解析。(内嵌js) 如果解析到 script 标签，渲染引擎会判断这是一段脚本，此时 HTML 解析器会暂停 DOM 的解析（因为 js 可能会修改生成的 DOM 结构）
    <html>
    <body>    
        <div>1</div>    
        <script>    
            let div1 = document.getElementsByTagName('div')[0]    
            div1.innerText = 'time'    
        </script>    
        <div>test</div></body></html>

    解析成的DOM树

    脚本执行完成之后，HTML 解析器恢复解析过程，继续解析后续内容，直到生成完整的 DOM。

    1. 阻塞 DOM 的解析 （外联js）
    <html>
    <body>    
    <div>1</div>    
    <script type="text/javascript" src='foo.js'></script>    
    <div>test</div></body></html>

    当执行到 js 时，需要先下载这段 js 代码，js 文件的下载过程中会阻塞 DOM 解析，下载一般都是耗时的，受网络环境、js 文件大小等因素。

    Chrome 的优化方式：预解析操作，渲染引擎收到字节流之后，开启一个预解析线程，来分析 HTML 文件中包含的 js、css 等文件，解析到文件之后，预解析线程会提前下载这些文件.

    我们可以使用使用 CDN 来加速 js 文件的加载，压缩js文件的体积。

    如果 js 文件中没有操作 DOM 相关的代码，可以将该 js 脚本设置为异步加载。通过 async 或 defer 来标记。

    # css阻塞

    link 引入的 css 文件，在执行 js 之前，需要先解析 js 语句之上所有的 css 样式。如果代码里引用了外部的 css 文件，需要在执行js之前，需要等待外部的 css 文件下载完成，并解析生成 CSSOM 对象之后，才能够执行 js 脚本。

    主要是因为 不知道 js 是否操作了 CSSOM。所以渲染引擎在遇到 js 引擎的时，不管是否操作了 CSSOM，都会执行 CSS 文件下载，解析操作。

    我们要重点关注 js 文件和样式表，js 阻塞 DOM，样式文件阻塞 js。一定程度上 css 也会阻塞 DOM。
