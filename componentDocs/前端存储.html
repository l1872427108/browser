    # 前端存储

    Cookie、SessionStorage、LocalStorage 都可以被用来在浏览器端存储数据。都是字符串类型的键值对。 (另一种存储模式叫：session，这种级别的存储属于服务器端级别的存储)。

    # Cookie

    Cookie 是纯文本格式，在 web 服务器和浏览器之间传递。

    Cookie 本质上属于 http 的范畴，因为 http 协议本身是无状态的，服务端是没有办法区分请求来自于哪个。所以我们需要 cookie 去维持客户端的状态。

    cookie 的生成方式：

    1. 客户端生成：document.cookie 属性。
    2. 服务端生成：web 服务器通过发送一个称为 Set-cookie 的值，可以从 document.cookie 中读取。
    3. HttpOnly: 为避免跨域脚本 XSS 攻击，通过 js 的document.cookie API 无法访问有 httpOnly 标记的 Cookie。

    Cookie 缺点：

    1. 安全性： cookie 在 HTTP 中明文传输，很容易被非法用户截获，然后进行一系列的串改，在 Cookie 的有效期内重新发送给服务器，这是相当危险的。另外，在 httpOnly 为 false的情况下， Cookie 能够直接被 js 脚本来读取。
    2. 大小限制： cookie 的大小限制在 4kb，不适用大量存储。
    3. 增加流量，每次请求都会被自动添加到 header 中，无形中增加了流量。不管地址下的某一个地址不需要 Cookie，都会携带上完整的 Cookie，随着请求数的增加，都会造成巨大的性能浪费。

    cookie 的属性有:

    1. http-only: 不能被客户端更改访问。
    2. Secure: 只允许在 https 下传输。
    3. Max-age: 失效的秒数
    4. expire: cookie 的最长有效时间。不设置，cookie 的生命周期与会话期相同。

    document.cookie('名字=数据;expire=时间')；

    把它的 Max-age 设置为 0，就可以失效，可就消失了。

    # Web Storage

    SessionStorage 和 LocalStorage 都是本地储存，空间要大。 5-10M

    浏览器提供了window下的两个属性

    var data = xxStorage.getItem('key');
xxx.setItem('key', value);
xxx.removeItem('key');
xxxStorage.clear();

    Storage 事件：

    Storage 对象发送变化时，发生创建/更新。删除数据项。在同一个页面内发生的改变不会起作用。 - key 删除的 key。 - newValue: 新设置的值。 - oldValue: 调用改变前的 value 值。 - url - storageArea

    # localStorage

    在关闭浏览器后数据以来存在，除非手动清除。

    相同浏览器下不同标签在同源情况下可以共享 localStorage

    1. 容量。 5M，相比于 cookie 的 4k大大增加了，这 5M 是针对于一个域名的。
    2. 只存在客户端。可以避免 Cookie 带来的性能和安全性问题。
    3. 接口封装。暴露在全局，可以非常方便的操作。

    利用 localStorage 的较大容量和持久特性，可以存储一些稳定的资源。 logo，base64格式的图片资源。 美团的红包卷就是利用了 localStorage + storage 事件。

    # SessionStorage

    关闭浏览器或标签页后失效。

    只在当前标签可用，当前标签的 iframe 中且同源可以共享。

    1. 容量, 5M.
    2. 只存在客户端
    3. 接口封装。

    是一个会话级别的存储。会画结束（界面关闭），就会消失。

    利用 SessionStorage 可以对表单信息进行维护，可以保证刷新页面也会存在。 可以利用它存储本次浏览记录，页面关闭就不需要这些记录了。微博就采取这样的存储。

    # session

    Session 代表着服务器和客户端一次会话的过程。Session 对象存储特定用户会话所需的属性及配置信息.当客户端关闭会话，或者 Session 超时失效时会话结束。

    1. 服务器端级别的存储, 保存在服务端。

    2. session 的运行依赖 sessionId，而sessionId 保存在 cookie 中，所以如果禁用掉 cookie，session 就不能够使用了，如果还想使用，可以把 sessionId 保存在 URL 中。

    3. session 一般用来跟踪用户的状态。

    4. session 的安全性更高，保存在服务端，不过一般为了使服务端性能更加，会考虑部分信息保存在 cookie 中。

    # 对比 session 和 cookie。

    1. cookie 存放在客户端浏览器上，session存放在服务器上。
    2. cookie 不是很安全，别人可以分析存放在本地的 cookie 进行 cookie 欺骗。安全性可以使用 session。
    3. session 会在一定时间内保存在服务器上，当访问增多，为了减轻服务端的压力，可以使用 cookie。
    4. session 存储数据多余 cookie。
    5. cookie 可以长时间保存， session 在客户端关闭或 session 超时都会失效。
    6. cookie 只能保存 ASCII，Session 可以存任意数据类型。

    # indexDB

    是浏览器提供的本地数据库，可以被网页脚本创建和操作。允许储存大量数据，提供查找接口，还能建立索引。 indexDB 不属于关系型数据库（不支持 SQL 查询语句）。

    1. 键值对储存
    2. 异步. IndexedDB 操作时不会锁死浏览器，用户依然可以进行其他操作
    3. 支持事务. 这意味着一系列操作步骤之中，只要有一步失败，整个事务就都取消，数据库回滚到事务发生之前的状态.
    4. 同源限制: IndexedDB 受到同源限制，每一个数据库对应创建它的域名.
    5. 存储空间大。 一般来说 250 MB。
    6. 支持二进制存储。 不仅可以储存字符串，还可以储存二进制数据 ArrayBuffer 和 Blob 对象。
    window.indexedDB.open(databaseName, version)。
