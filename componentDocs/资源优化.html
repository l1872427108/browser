    # 资源优化

    1. 减少 http 请求数量，资源越多，在网络上造成的开销就越大。
    2. 减少请求资源大小。

    # 图片优化

    1. 图片格式的选择, 找一个合适的格式。

      • jpg/jpeg: 有损压缩的图片，色彩感也不错，色彩高达16000多种，可以满足我们开发的需要。但是不适用一些纹理较强的图片。
      • png: 可以适用于一些纹理性，边角清晰的使用。但是体积要更大，色彩感丰富，适用于一些小店的图标。
      • webP: 跟 png 的质量有的一比，提及确实比 png 更小。Google 提倡使用。 兼容性存在一些小问题。 其实三种都可以，质量方面我们很难看出来区别，所以还是要选择提及小一点的。

    2. 图片加载顺序，注重图片优先级，先把重要的加载进来。

    3. 图片的懒加载。

      • 原生懒加载 img 上添加属性 loading="lazy"
      • 借助一些第三方库。 verlok/lazyload yall.js blazy.

    4. 渐进式图片。 JPEG 本身的特点

      • Baseline JPEG 自身而下的加载图片
      • Progressive JPEG 模糊到清除. 用户体验更好。

    5. 响应式图片。 h5 新添加的属性来可以根据不同的尺寸来选择图片尺寸。 我们尽量不要将大的图片资源，为了在小屏幕上适应而缩小图片大小，这样在小屏幕上加载一个很大的图片，会造成很大的资源浪费。

    6. iconfont

      • 一套字体，减少获取时的请求数量和体积。
      • 矢量图形，可伸缩.
      • 直接通过 css 修改样式（颜色、大小）
      • iconfont 不能多颜色。
      • 语义感没有。

    7. svg

      • 保持了图片能力，支持多色彩。
      • 独立的矢量图形。体积更小，不用引入一套。
      • XML 语法。 SEO/读屏软件读取。

    # 字体优化

    字体未下载完成时，浏览器隐藏或自动降级，导致字体闪动。

    1. FOIT: 文字从看不到到看到
    2. FOUT: 没有经过样式渲染，呈现两种字体。

    当我们网络下载字体，就会遇到这种问题。为了尽早的显示字体，我们需要进行一些调整。

    1. font-display

      • block: 3秒等待（不显示）加载完就替换字体，否则就默认字体。
      • swap 上来默认字体， 直到需要的字体 (很快的看到文字)
      • fallback 100ms 没有下载完, 就是用默认，下载完就是用下载好的字体。
      • optional: 浏览器经过100ms判断，如果下载的字体还很慢，就不使用，使用默认字体。就不会做改变了。

    2. AJAX + Base64 嵌入到 js/css 兼容性好，通过载入文件里面。但是没有办法控制文字的缓存。

    # gzip

    对传输资源进行体积压缩，压缩率可高达 90%. 通过 nginx 来开启 gzip。

    gzip on 开启压缩 
gzip_min_length 1k 最少1k采取压缩
gzip_comp_level 6 压缩登记  1～9 对性能消耗
对文本压缩。 图片一般不压缩
gzip_static on 对已经压缩的静态资源直接利用
gzip_vary on;  在响应头部添加一个 vary 属性，告诉浏览器端提供了 gzip
gzip_buffers 优化压缩过程。
gzip_http_version 1.1 http版本

    # keep alive 对 TCP 链接请求复用

    Initial connection 就是对 TCP 链接的链接。 
connection: keep-alive。
keepalive_timeout 0; 就是关闭。
keepalive-timeout 65; 超时的时间。 65s 都没有使用就会关闭
keepalive_requests 100;  建立 TCP 一共可以发送多少个请求。

    # 资源加载顺序

    preload, prefetch 调整优先级.

    每一个资源都有一个 priority 属性，代表的是资源的优先级。

    1. preload:

      • 提前加载较晚出现，但对当前页面非常重要的资源。
      • preload 只负责加载，不负责解析。

    2. prefetch:

      • 提前加载资源，但是不是对当前页面有用的。
      • 这样子当其他页面用到，会从 prefetch cache 去获取缓存。

    # 预渲染

    1. 大型单页应用的性能瓶颈： JS 下载 + 解析 + 执行。
    2. SSR 主要的问题：权衡的一种方式，牺牲 TTFB 来补救 First Paint TTFB 请求后台，包括从后台拿过来的时间，包括后台处理的时间。传输的过程中也会增加。实现复杂。
    3. 预渲染 打包时提前渲染页面，没有服务端的参与。 打包的时候提前一个渲染。但是没有 SSR 那么 完美。

    # windowing(窗口化)提高列表性能

    只给用户能够看到的表格或列。

    不创建无限多的元素，复用每一行的组件，把不使用的回收，再去创建。

    作用： 加载大列表、大表单的每一行严重影响性能。

    懒加载会让 DOM 变的过大。

    windowing 只渲染可见的行，渲染和滚动的性能都会提升。

    # 骨架屏

    减少布局移动。

    占位。加载完之后再去渲染, 提高用户感知性能。
